## 一、市场背景与行业判断

### 1.1 结构性矛盾

当前数据分析领域存在一个结构性矛盾：**数据量指数级增长 vs 数据人才供给线性增长**。

- 企业 **80%** 的数据分析需求停留在"描述性分析"（发生了什么）
- 真正产生价值的"诊断性分析"（为什么发生）和"预测性分析"（将会怎样）因门槛过高，只有少数企业能触及

### 1.2 核心判断

> 数据分析 Agent 不是替代数据分析师，而是把"会用 SQL 的分析师"的能力赋予"只会问问题的业务人员"。

### 1.3 市场规模估算

| 维度 | 估算 |
|------|------|
| **TAM（全球）** | 企业分析市场约 $300B+（Gartner 2025） |
| **SAM（中国）** | 中国 BI + 数据分析工具市场约 ¥200-300亿 |
| **SOM（可触达）** | 聚焦中大型企业业务部门 + 中小企业管理层，约 ¥20-50亿 |

## 二、用户画像

### 2.1 核心用户分层

| 层级 | 角色 | 数据能力 | 核心诉求 | 付费意愿 |
|------|------|---------|---------|---------|
| **P0** | 业务运营/市场人员 | 零 SQL，靠 Excel 和"求分析师" | 快速拿到数据结论，不求人 | ★★★★ |
| **P0** | 中小企业管理者 | 看 Dashboard，不懂建模 | 经营异常预警，决策有据可依 | ★★★★★ |
| **P1** | 初中级数据分析师 | 会 SQL，写得慢 | 提效 80%，从取数中解放 | ★★★ |
| **P2** | 产品/增长负责人 | 能写简单查询 | 自助式探索分析，快速验证假设 | ★★★ |

### 2.2 典型用户故事

#### 故事一：张经理（连锁餐饮区域负责人）

> 每周一要看 20 家门店的经营数据，目前靠总部数据团队周三才能出报告，错过了最佳决策窗口。他需要的是——"帮我看看华东区哪些门店上周营收环比下降超过 15%，可能是什么原因"——这样一句话就能得到答案。

#### 故事二：李总监（电商运营总监）

> 大促结束后需要快速复盘，但数据散落在多个系统（订单、流量、客服、仓储）。目前需要 3 个人花 2 天整理数据。他希望："帮我做一份 618 大促复盘报告，对比去年同期，重点分析转化率变化的原因。"

#### 故事三：王分析师（初级数据分析师）

> 每天接 10+ 个取数需求，80% 时间在写 SQL 和整理 Excel，真正做分析的时间不到 20%。他希望把简单取需求交给 Agent，自己专注深度分析。

## 三、痛点分析与伪需求鉴别

### 3.1 真需求（有明确场景 + 愿意付费）

| 痛点 | 真伪判断 | 验证逻辑 |
|------|---------|---------|
| **"取数慢"——业务提需求到拿到数据平均 3-5 天** | ✅ 真需求 | 存在明确的时间成本浪费，企业愿意为效率付费 |
| **"看不懂"——有 Dashboard 但不知道数据意味着什么** | ✅ 真需求 | BI 工具普及率高但使用深度低，说明"看懂"是瓶颈 |
| **"问不出"——知道数据里有答案，但不会写 SQL/不会建模** | ✅ 真需求 | NLP-to-SQL 技术成熟度已验证，用户转化率可量化 |
| **"汇报难"——从数据到 PPT/报告需要大量手工整理** | ✅ 真需求 | 高频 + 重复 + 耗时，自动化 ROI 明确 |

### 3.2 伪需求（需警惕）

| 痛点 | 真伪判断 | 验证逻辑 |
|------|---------|---------|
| **"我需要 AI 帮我做战略决策"** | ⚠️ 伪需求 | 战略决策需要 context（行业 know-how、竞对信息、组织能力），纯数据分析 Agent 无法覆盖；且决策责任不可转移 |
| **"替代数据分析师"** | ⚠️ 伪需求 | 数据治理、口径定义、异常排查等需要人类专家判断；Agent 是放大器，不是替代品 |
| **"全自动数据清洗"** | ⚠️ 半伪需求 | 清洗规则与业务强相关，全自动不现实；但"辅助清洗 + 确认"可行 |

### 3.3 需求优先级矩阵

```
紧急度 ↑
  │
高│  [P0] 取数自动化        [P0] 异常检测预警
  │
中│  [P1] 归因分析           [P1] 报告自动生成
  │
低│  [P2] 预测分析           [P2] 数据治理辅助
  │
  └──────────────────────────────────────→ 价值度
        低                  中                高
```

## 四、需求调研与立项建议

### 4.1 调研方法论

| 阶段 | 方法 | 目标 | 周期 |
|------|------|------|------|
| **定性** | 深度访谈 15-20 人（覆盖 P0/P1 用户） | 验证痛点真实性，发现未被满足的需求 | 2 周 |
| **定量** | 问卷 500+ 样本 | 量化痛点优先级，估算 TAM/SAM/SOM | 1 周 |
| **竞品** | 体验 5-8 款竞品 + 用户口碑分析 | 找到差异化切入点 | 1 周 |
| **MVP 验证** | 用现有 LLM + 简单 Pipeline 做 Demo | 验证核心假设"用户愿意用自然语言做分析" | 2 周 |

### 4.2 调研问题清单（深度访谈参考）

**业务人员（P0）：**
1. 你每周花多少时间在"等数据"或"找数据"上？
2. 上次需要数据支持决策时，流程是怎样的？花了多久？
3. 你用过哪些数据工具？最大的障碍是什么？
4. 如果有一个工具能用自然语言查询数据，你最想问什么问题？
5. 你愿意为这样的工具每月付多少钱？

**数据分析师（P1）：**
1. 你每周处理的取数需求中，简单查询占比多少？
2. 哪些环节最耗时？（写 SQL、数据清洗、报告整理？）
3. 如果简单需求被自动化，你省下的时间会用来做什么？
4. 你对 AI 生成的 SQL 的信任度如何？需要怎样的验证机制？

### 6.2 AI 方案设计

#### 核心技术路线

| 模块 | 技术选型 | 原因 |
|------|---------|------|
| **意图理解** | Fine-tuned LLM + 意图分类器 | 需要区分"查询型"、"分析型"、"探索型"三类意图 |
| **NL2SQL** | 多轮修正的 Text2SQL（非单次生成） | 单次生成准确率不够，需要"生成→执行→验证→修正"循环 |
| **语义层** | 指标字典 + 实体链接 | 解决"上个月"到底是自然月还是近 30 天的歧义 |
| **归因分析** | LLM + 统计方法（SHAP/相关性） | LLM 擅长归因假设，统计方法负责验证 |
| **可视化** | LLM 选择图表类型 + ECharts/Vega 渲染 | 让 Agent 决定用柱状图还是折线图，而非用户手动选 |
| **报告生成** | 结构化模板 + LLM 填充 | 保证格式一致性，同时保持叙述灵活性 |

#### 关键技术挑战与解法

**挑战一：NL2SQL 准确率**

| 维度 | 说明 |
|------|------|
| 当前业界水平 | 简单查询 85%+，复杂查询 60-70% |
| 解法 | ① 业务语义层减少歧义 ② 多轮修正机制 ③ 人工确认兜底 |
| 目标 | 简单查询 95%+，复杂查询 85%+ |

**挑战二：数据分析的"正确性"**

| 维度 | 说明 |
|------|------|
| 风险 | LLM 可能生成统计上错误的结论 |
| 解法 | ① 分析结果必须可追溯到具体数据 ② 统计校验层 ③ 置信度标注 |

**挑战三：企业数据安全**

| 维度 | 说明 |
|------|------|
| 风险 | 数据不能出企业边界 |
| 解法 | ① 私有化部署选项 ② 数据脱敏 ③ 审计日志 |

### 6.3 Agent 工作流详解

```
用户输入："上个月华东区业绩为什么下降了？"

Step 1 - 理解 Agent
  ├── 意图识别：诊断型分析（WHY）
  ├── 实体提取：时间=上个月，区域=华东区，指标=业绩
  ├── 语义解析：上个月 → 2026年6月（自然月），业绩 → GMV（根据指标字典）
  └── 输出：结构化查询意图

Step 2 - 分析 Agent
  ├── 数据获取：生成SQL，查询华东区6月GMV及同比/环比
  ├── 异常定位：GMV环比下降12%，主要拖累是A、B、C三家门店
  ├── 归因分析：
  │   ├── 维度下钻：按门店→品类→时段逐层分析
  │   ├── 外部因素：检查是否有竞对活动、天气、节假日等
  │   └── 关联分析：客单价 vs 订单量 vs 转化率
  └── 输出：归因结论 + 置信度

Step 3 - 表达 Agent
  ├── 结论叙述：自然语言描述分析结果
  ├── 可视化：选择合适的图表（趋势图+门店对比图）
  ├── 建议生成：基于归因给出可执行建议
  └── 输出：完整的分析报告
```

## 七、产品差异化亮点

### 亮点一："数据故事"而非"数据图表"

> 竞品给你一张图表，我们给你一个故事。

**传统 BI 流程：**
```
用户问：营收怎么样？
→ 返回一张折线图
→ 用户自己看：哦，下降了
→ 用户自己猜：为什么下降？
→ 用户再问分析师：帮我查一下原因
→ 等 2 天...
```

**我们的体验：**
```
用户问：上个月华东区业绩为什么下降了？

Agent 回复：
📊 概况：华东区6月GMV为1.2亿，环比下降12%，同比持平。

🔍 主要原因（按影响度排序）：
1. A门店（影响占比45%）：竞对"XX品牌"6月在周边新开业，
   导致该店客流环比下降23%。
2. B门店（影响占比30%）：5月大促透支了6月需求，
   复购率环比下降18%。
3. C门店（影响占比15%）：供应链问题导致SKU缺货率升至23%。

📈 趋势图：[自动生成的门店维度对比图]

💡 建议：
· A门店：加强会员粘性运营，推出专属优惠对抗竞对
· B门店：控制促销节奏，增加6月独立活动
· C门店：优先解决供应链，预计恢复周期2周
```

**技术实现：** Multi-Agent 协作——数据 Agent 定位异常 + 归因 Agent 分析原因 + 策略 Agent 生成建议

### 亮点二："业务记忆"——越用越懂你

| 阶段 | 体验 |
|------|------|
| 第 1 次对话 | 你问"上个月业绩怎么样"，它给你全局概览 |
| 第 5 次对话 | 它知道你关心华东区、关注 GMV 和毛利率 |
| 第 10 次对话 | 它主动告诉你"A门店本周异常，建议关注" |
| 第 30 次对话 | 它了解你的汇报习惯，自动生成你格式的周报 |

**技术实现：**
- 用户画像持久化（关注的指标、常用维度、汇报习惯）
- 对话历史摘要（跨 session 的上下文记忆）
- 主动推送机制（异常检测 → 主动通知，而非等用户来问）

### 亮点三："分析过程透明化"——白盒而非黑盒

> 用户可以展开查看 Agent 的完整推理链路：用了哪些表、写了什么 SQL、做了什么统计检验。

```
[点击展开查看分析过程]

📋 执行步骤：
1. 查询了 orders 表，获取华东区6月订单数据（SQL: SELECT ...）
2. 计算环比变化，发现整体下降12%
3. 按门店维度下钻，定位A、B、C三家为主要拖累
4. 查询 competitors 表，发现A门店周边6月新增竞对门店1家
5. 查询 campaigns 表，发现B门店5月参加了大促活动
6. 查询 inventory 表，发现C门店6月缺货率23%

⚠️ 注意事项：
· 归因分析中"竞对影响"的置信度为中等（基于地理位置推断，
  未获取竞对实际销售数据）
· 建议结合实地调研进一步验证
```

**价值：** 数据分析师可以审查和修正，管理者可以建立信任，满足合规审计需求

### 亮点四："一键出报告"——从数据到交付物

> 对话结束 → 一键生成 PPT/Word/邮件报告 → 自动排版 + 数据可视化 + 结论叙述

**价值：** 解决"最后一公里"问题——分析做完了，但整理成汇报材料还要半天

支持格式：
- PDF 报告（适合邮件发送）
- PowerPoint（适合会议汇报）
- 飞书/钉钉文档（适合团队协作）
- 邮件摘要（适合定期推送）

### 亮点五："多人协作分析"——不是一个人在战斗

> 运营问了一个问题 → Agent 给出初步结论 → 数据分析师补充验证 → 产品经理加入讨论 → 自动生成会议纪要和 Action Items

**价值：** 打破"分析师在群里丢个 SQL 截图"的低效协作模式

## 八、产品定义与 PRD 框架

### 8.1 产品定位

> **一句话：** 让每个人都能用对话的方式，从数据中获得可行动的洞察。

| 维度 | 定义 |
|------|------|
| **产品名称** | DataInsight Agent（暂定） |
| **目标市场** | 中国中大型企业（500人+）的业务部门 + 中小企业（50-500人）的管理层 |
| **核心价值** | 从"提需求等 3 天"到"问一句话即刻得答案" |
| **商业模式** | SaaS 订阅（基础版/专业版/企业版）+ 私有化部署 |

### 8.2 用户故事地图

```
用户目标：

  语义解析      对比分析      订阅推送     审计日志
```

### 8.3 功能规格（P0 功能）

#### F01：自然语言查询

| 项目 | 说明 |
|------|------|
| **功能描述** | 用户输入自然语言，系统自动转换为 SQL 并返回结果 |
| **输入** | 自然语言文本（支持中英文混合） |
| **输出** | 查询结果表格 + 自动生成的可视化图表 |
| **业务规则** | ① 支持多轮追问 ② 自动识别时间/维度/指标 ③ 超时自动降级 |
| **异常处理** | 无法理解时给出澄清问题，而非错误提示 |

#### F02：指标字典

| 项目 | 说明 |
|------|------|
| **功能描述** | 配置业务指标的定义、计算公式、数据来源 |
| **输入** | 指标名称、计算公式、关联表/字段、业务口径说明 |
| **输出** | 语义解析时的指标映射 |
| **业务规则** | ① 支持别名（"营收"="GMV"="销售额"） ② 支持派生指标 ③ 版本管理 |

#### F03：归因分析

| 项目 | 说明 |
|------|------|
| **功能描述** | 当指标出现异常波动时，自动进行多维度下钻分析 |
| **输入** | 异常指标 + 时间范围 |
| **输出** | 归因结论（按影响度排序）+ 置信度 + 建议 |
| **业务规则** | ① 支持自定义下钻维度 ② 最多下钻 5 层 ③ 每个结论需可追溯到具体数据 |

#### F04：报告自动生成

| 项目 | 说明 |
|------|------|
| **功能描述** | 将分析对话转化为格式化的报告 |
| **输入** | 对话记录 + 报告模板 |
| **输出** | PDF/PPT/Word/在线文档 |
| **业务规则** | ① 支持自定义模板 ② 图表自动嵌入 ③ 支持手动编辑 |

## 十一、风险与对策

| 风险 | 概率 | 影响 | 对策 |
|------|------|------|------|
| NL2SQL 准确率不达标 | 高 | 高 | 多轮修正 + 人工兜底 + 渐进式开放复杂查询 |
| 企业数据安全顾虑 | 高 | 高 | SOC2 认证 + 私有化部署 + 数据不出域承诺 |
| LLM 成本过高 | 中 | 中 | 小模型处理简单查询，大模型处理复杂分析；缓存高频查询 |
| 用户不愿改变习惯 | 中 | 中 | 从"最痛的场景"切入，不求全面替代，先做增量 |
| 竞品快速跟进 | 中 | 中 | 靠"业务记忆"和"数据故事"构建差异化壁垒 |
| 数据质量问题 | 高 | 中 | 数据质量检测 + 异常数据提示 + 不强制给出结论 |

## 十二、总结与建议

### 12.1 可行性判断：✅ 可行，但需聚焦

**核心逻辑：**

1. **需求真实**——"取数慢、看不懂、汇报难"是经过验证的企业痛点
2. **技术可行**——NL2SQL + LLM 的技术栈已基本成熟，关键是工程化和产品化
3. **市场有空间**——现有竞品要么太贵（ThoughtSpot）、要么太浅（简单 Text2SQL）、要么太重（传统 BI）
4. **差异化有机会**——"数据故事" + "业务记忆" + "白盒分析"是当前竞品未覆盖的组合

### 12.2 关键建议

| 建议 | 说明 |
|------|------|
| **先做窄再做宽** | MVP 聚焦 1-2 个行业（如零售/电商），积累业务语义层，再横向扩展 |
| **先做"最后一公里"** | 不要从头建 BI，而是接入现有数据源，做"数据→洞察→报告"的最后一段 |
| **先赢信任再谈替代** | 定位为分析师的"超级助手"，而非替代品，降低用户心理阻力 |
| **重视数据治理前置** | 没有好的数据字典和指标定义，再强的 Agent 也是垃圾进垃圾出 |
| **快速验证核心假设** | 2 周内用现有 LLM 做出 Demo，找 10 个目标用户体验，验证"自然语言查数据"的接受度 |
